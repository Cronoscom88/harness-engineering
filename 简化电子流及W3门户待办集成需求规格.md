# 应用系统简化电子流及W3门户待办集成需求规格
## 一、简化电子流需求
### 1.1 待办任务自动推送
问卷投放操作完成后，系统需自动向所有接收问卷的用户推送待办任务，保障用户及时接收问卷作答通知，无推送遗漏问题。

### 1.2 待办与问卷直接联动
用户点击W3门户待办任务条目后，需直接跳转至对应问卷的作答页面，无需经过中间跳转环节，实现待办任务与问卷作答的无缝衔接，提升操作效率。

### 1.3 作答完成后状态同步消除
用户完成问卷作答并提交后，系统需自动关闭问卷作答页面，同时触发待办任务消除逻辑，将该用户对应的此问卷待办任务从待办列表中移除，确保待办状态与实际业务进度一致。

### 1.4 多问卷待办灵活处理
1.  支持**单问卷多用户投放**：一个问卷可批量向多个用户投放，每个接收用户均会收到专属的问卷作答待办任务。
2.  支持**单用户多待办并行**：一个用户可同时接收多个不同问卷的待办任务，可在待办列表中自主选择不同问卷，点击后跳转至对应作答页面，各问卷作答流程相互独立、互不干扰。

## 二、W3门户待办集成需求
### 2.1 集成背景与目标
W3门户为公司内部统一员工门户，是员工查看公司各类信息、集中处理各系统待办任务的核心入口。为提升用户体验，确保本应用系统的问卷作答待办能被用户及时查看和处理，需**预留与W3门户待办的集成接口**，便于后续在公司环境部署后，快速完成对接，实现待办任务的统一展示、处理与状态同步。

### 2.2 集成接口清单
本应用系统需按要求预留W3门户待办集成接口，接口请求方式及核心功能如下表所示：

|接口名称|请求方式|功能说明|
|---|---|---|
|新增待办|POST|问卷投放后，向W3门户推送新的问卷作答待办任务|
|删除待办|POST|用户提交问卷后，删除W3门户中对应的待办任务|
|更新待办|DELETE|待办任务信息（如描述、状态）变更时，更新W3门户中对应待办数据|
|查询我的待办|GET|用户在W3门户查看个人待办时，获取本系统推送的问卷作答待办|
|申请待办查询|GET|查询用户作为问卷投放人发起的相关待办任务记录|
|历史待办查询|GET|查询用户已完成提交的问卷作答待办历史记录|

### 2.3 待办接口字段规范
接口调用时需严格遵循以下字段要求传参，确保与W3门户待办系统正常对接，字段说明如下：

|字段名|必填项|字段说明|格式要求|
|:---|:---|:---|:---|
|**appName**|是|应用系统名称，用于W3门户区分不同系统的待办任务|填写本应用系统正式名称，无特殊格式|
|**appURL**|否|移动端待办推送传参，传值则将业务链接推送至WeLink待办，不传则推W3门户缺省待办页面|符合URL标准格式|
|**type**|否|任务类型标识|统一填写固定值：**J2EE**|
|**taskDesc**|是|待办任务描述，需清晰展示核心信息|建议包含「问卷标题+业务流程+状态」，示例：【问卷作答】员工年度满意度调研，待作答|
|**taskURL**|是|W3门户「我的待办」详情跳转地址|必须为**绝对地址**，符合URL规范|
|**taskUUID**|是|待办任务全局唯一标识|全局唯一，不可重复，建议采用UUID生成规则|
|**time**|是|待办任务创建时间|符合时间格式：**yyyy-MM-dd HH:mm:ss**|
|**handler**|是|待办处理人（问卷接收人）|格式为「姓名 工号」，示例：zangsan 00112356|
|**applicant**|否|待办申请人（问卷投放人）|格式为「姓名 工号」，示例：zangsan 00112356|
|**reserve3**|否|W3门户「我的申请」详情跳转地址|必须为**绝对地址**，符合URL规范|
|**reserve10**|否|备用扩展字段|无特殊格式要求，按需填充|
|**reserve2**|否|备用扩展字段|无特殊格式要求，按需填充|
