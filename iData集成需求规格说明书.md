# iData集成需求规格说明书
## 一、引言
### 1.1 编写目的
明确当前开发系统与iData主数据系统集成的要求，确保系统能预留标准化接口，实现人员基本信息和部门组织数据的实时同步，保障数据一致性。

### 1.2 术语定义
- **iData**：公司主数据系统，提供人员及部门组织的主数据管理服务
- **MQS消息服务**：基于消息队列（Message Queue）的异步数据同步服务
- **增量同步**：仅推送变更数据
- **全量同步**：推送全部数据
- **Topic**：消息队列中的主题，用于分类消息

### 1.3 文档范围
本规格说明书涵盖与iData系统集成相关的接口定义、数据模型、同步机制及非功能需求。

---

## 二、综合描述
### 2.1 系统功能
当前开发系统需通过MQ消息集成方式，从iData实时获取人员基本信息和部门组织信息，确保本系统数据与主数据源保持一致。

### 2.2 用户类型与特征
- 系统管理员：负责配置集成参数、监控数据同步状态
- 数据运维人员：处理同步异常，审核数据映射规则
- 开发人员：实现集成接口和数据处理逻辑

### 2.3 运行环境
- 消息中间件：支持MQ协议（如IBM MQ、RabbitMQ等），需确保与iData的消息中间件兼容
- 网络要求：确保系统与iData网络连通性，支持TLS加密传输（可选）
- 软件环境：当前系统需集成MQ客户端库，支持JSON消息解析

## 三、系统集成需求
### 3.1 集成架构
采用发布/订阅模式，iData作为消息发布方，当前系统作为订阅方，通过MQ队列实现数据异步同步。

### 3.2 MQ消息服务接口
#### 3.2.1 人员基本信息
- 增量Topic：`T_HW_HR_ALL_PEOPLE_INF_TOPIC`
- 增量推送频率：每天5:00和17:00各推送一次
- 全量Topic：无
- 对象：人员
- 数据使用说明：主键为 `employee_number + language + tenantId`
- 消息格式：JSON

```json
{
   "employeeNumber": "E12345",
   "sex": "M",
   "lastName": "Smith",
   "firstName": "John",
   "fullName": "John Smith",
   "oldnumber": "E67890",
   "englishName": "John Smith",
   "startDate": "2020-01-01",
   "effectiveStartDate": "2023-03-15",
   "effectiveEndDate": "2099-12-31",
   // ...其他字段待补充
}
```


#### 3.2.2 部门组织信息
- 增量Topic：`T_IDATA_INC_ORGANIZATION`
- 增量推送频率：每小时一次
- 全量Topic：`T_IDATA_ALL_ORGANIZATION`
- 全量推送频率：不定时下发
- 对象：组织
- 数据使用说明：默认只下发 `public_flag='Y'` 的数据，主键为 `deptCode + language + tenantId`
- 消息格式：JSON

```json

   "coaCombination": "100-200-300",
```

### 3.3 数据同步机制
#### 3.3.1 增量同步
- 系统实时监听MQ Topic，接收增量数据后更新本地数据库
- 人员信息：每天5:00和17:00接收增量数据
- 部门组织：每小时接收增量数据

#### 3.3.2 全量同步（仅部门组织）
- 首次部署时，通过iData管理界面手动触发全量数据同步
- 或由iData系统不定时下发全量数据
- 全量同步时，系统需根据 `public_flag='Y'` 过滤数据
- 基于主键 `deptCode + language + tenantId` 覆盖本地数据

#### 3.3.3 数据初始化策略
- 若首次部署时未获取全量数据，需通过连续接收增量数据逐步完成初始化
- 或通过其他方式（如数据导入工具）补充初始数据

#### 3.3.4 异常处理
- 消息消费失败时，需将消息重新入队或记录错误日志，支持手动重试
- 网络中断时，MQ需确保消息持久化，恢复后重新投递
- 消息处理失败时记录详细错误信息，便于排查问题

### 3.4 消息校验与映射
#### 3.4.1 人员基本信息校验
- 接收消息后需验证主键唯一性，避免重复数据
- 验证必填字段是否完整
- 校验数据格式是否符合要求

#### 3.4.2 部门组织信息校验
- 接收消息后需验证 `public_flag='Y'`，过滤无效数据
- 验证主键 `deptCode + language + tenantId` 的唯一性
- 校验部门层级关系是否合理

#### 3.4.3 字段映射规则
- 建立iData字段与当前系统字段的映射关系
- 处理字段命名差异（如`sex`→`gender`）
- 定义数据转换规则

### 3.5 集成接口配置管理
- 系统需提供管理界面，支持配置MQ连接参数（地址、端口、认证信息）
- 支持订阅Topic的配置管理
- 提供消息解析规则的配置界面
- 支持集成状态的监控和报警配置

## 四、非功能需求
### 4.1 性能需求
- 消息处理延迟：接收消息后需在5秒内完成解析与入库
- 高并发支持：支持多线程消费消息，确保大数据量同步不阻塞
- 吞吐量要求：每秒至少处理100条消息

### 4.2 安全性需求
- 认证：MQ连接需支持用户名/密码认证或TLS证书认证
- 数据保密：敏感字段在传输中加密（若iData提供）
- 权限控制：仅允许授权系统访问集成Topic
- 审计日志：记录所有数据访问和操作日志

### 4.3 可靠性
- 消息持久化：MQ需确保消息不丢失
- 幂等性：系统需设计幂等处理机制，防止重复消息导致数据错误
- 故障恢复：系统需支持自动恢复和手动恢复机制

### 4.4 可维护性
- 日志记录：记录消息接收时间、处理结果、错误详情
- 监控报警：当消息积压超过阈值或同步失败时发送报警
- 配置管理：支持动态配置调整，无需重启服务

### 4.5 可扩展性
- 支持后续新增iData其他数据主题的集成扩展
- 支持不同版本的接口兼容
- 支持多租户环境下的数据隔离
