# 问卷调查系统详细方案设计文档

## 1. 文档目标与范围

本方案用于定义一个企业级问卷调查系统的完整版本设计，满足以下目标：
- 本地可运行（开发/演示环境），后续平滑云化部署。
- 对齐既有规范：技术架构、安全红线、接口集成、UI视觉规范。
- 覆盖全链路能力：问卷设计、投放管理、填写回收、统计分析、导入导出、客户管理、权限与审计。

不在本阶段实现内容：
- 第三方支付、外部营销自动化闭环。
- 复杂 AI 文本分析（可作为后续增强）。

## 2. 需求清单（确认版）

### 2.1 核心功能
- 内部单点登录（SSO），本地运行时提供演示账号登录。
- 支持匿名填写。
- 跳题逻辑（条件分支）。
- 截止时间/次数限制。
- 防重复提交（账号、设备指纹、IP、Cookie 替代方案）。
- 结果导出 Excel/CSV。
- 统计图表展示。
- 简洁模式开关（填写页可切换精简 UI）。
- 问卷创建方式：
  - 快速创建（从空白模板和题库向导）。
  - Excel 导入创建。
  - 从系统模板创建。
- 投放管理（链接、批次、渠道、状态）。
- 客户管理（客户名单列表、客户名单模板）。

### 2.2 题型支持
- 单选题
- 多选题
- 评分题（Likert/星级）
- 矩阵单选题
- 矩阵多选题
- 主观题（文本）
- 过渡语（说明页/章节分隔）
- NPS 题（0-10 推荐值）

## 3. 架构与技术选型

> 基于已有规范中“Spring Boot + 微服务架构”为核心，并结合前端红线中 Vue Router + Axios 约束，采用 Vue3 前端。

### 3.1 总体架构（分层）
- 表现层：Vue3 + Vite + Element Plus（管理端与填写端）。
- 网关层：Spring Cloud Gateway（统一鉴权、路由、限流、审计入口）。
- 业务服务层（微服务）：
  - survey-service（问卷设计与题目管理）
  - delivery-service（投放与回收规则）
  - response-service（答卷提交与反作弊）
  - customer-service（客户名单与模板）
  - analytics-service（统计报表与导出）
  - auth-service（本地演示登录与 SSO 适配）
- 数据层：
  - PostgreSQL（主存储，问卷配置+业务数据）
  - Redis（缓存、限流、幂等、短期会话）
  - 对象存储（本地 MinIO，云上 OSS/S3）用于导入导出文件
- 事件层：
  - Kafka（答卷事件、导出任务、统计异步计算）

### 3.2 为什么这样选
- 满足规范中的核心框架要求（Spring Boot 微服务）。
- 支持本地到云化的一致开发模型（容器化、服务拆分、异步化）。
- 可扩展：后续可将统计分析沉淀到 ClickHouse，不影响主链路。

### 3.3 前端技术方案
- 框架：Vue3 + Vite + TypeScript + Element Plus。
- 路由：Vue Router 统一管理。
- 请求：Axios 统一封装 API，页面不直接写接口绝对路径。
- 图表：ECharts（NPS 分布、题型分布、趋势图）。
- UI 规范：
  - 主色：#0067D1 及梯度色板。
  - 字体：Windows 使用 Microsoft YaHei/Arial。

## 4. 模块设计

### 4.1 问卷设计中心
- 问卷基础信息：名称、描述、封面、状态、版本。
- 题目管理：增删改排、分组、必填、校验规则。
- 逻辑引擎：
  - 条件表达式：`if(questionA == "A") -> jump to questionC`
  - 支持多条件与优先级。
  - 防止环路跳转（发布前图结构校验）。
- 模板机制：
  - 系统模板（官方）
  - 企业模板（组织内沉淀）

### 4.2 投放管理
- 发布渠道：公开链接、定向名单、内部系统嵌入。
- 发布控制：开始时间、截止时间、总次数、单人次数。
- 批次管理：不同渠道生成不同批次码，便于归因统计。
- 关闭机制：手动停止/自动到期停止。

### 4.3 填写与回收
- 登录态填写（SSO 用户）与匿名填写并行。
- 简洁模式：隐藏非关键装饰信息，仅保留题干和选项。
- 自动保存草稿（可配置开关，匿名可基于本地 token）。
- 提交校验：必填、逻辑完整、频控、防重复规则。

### 4.4 客户管理
- 客户名单列表：客户 ID、名称、标签、联系信息、状态。
- 客户名单模板：字段模板、导入映射规则、必填校验。
- 名单导入：Excel/CSV，支持字段预览和错误行回传。

### 4.5 统计分析
- 实时看板：回收率、完成率、有效答卷数。
- 题型维度分析：
  - 选择题分布
  - 评分题均值/中位数
  - NPS（Promoter/Passive/Detractor）
  - 矩阵题热力分布
- 时间维度：按天/周/月趋势。
- 导出：明细数据 CSV、统计报表 Excel。

## 5. 数据模型（核心表）

### 5.1 主数据表
- `survey`：问卷主表（状态、版本、发布配置快照）
- `survey_question`：题目定义（题型、顺序、配置 JSON）
- `survey_logic_rule`：跳题规则
- `survey_publish_batch`：投放批次与渠道
- `survey_response`：答卷主表（提交人、匿名标识、提交时间）
- `survey_answer`：答案明细（按题存储）
- `customer_list`：客户名单主表
- `customer_template`：客户模板定义
- `import_task`：导入任务状态
- `export_task`：导出任务状态

### 5.2 关键设计点
- 题目配置采用 JSON 字段，兼容多题型扩展。
- 答案分为结构化列 + JSON 扩展字段，兼顾查询与灵活性。
- 所有业务表带审计字段：`created_by`, `created_at`, `updated_by`, `updated_at`。

## 6. 接口设计规范

### 6.1 API 设计原则
- RESTful 风格，统一前缀：`/api/v1/...`
- 统一响应结构：`code`, `message`, `data`, `traceId`
- 幂等要求：提交与导入导出任务需支持幂等键。

### 6.2 认证与集成
- 生产：对接 iDaaS OAuth2.0 授权码模式，网关层完成 token 校验。
- 本地：演示账号（admin/demo），保留与生产一致的认证接口结构。
- 对接 iData/HEDS 时遵从 AppKey/AKSK 与签名规范（按环境开关）。

### 6.3 禁止事项（落地）
- 不提供未公开接口。
- 不使用零散路径，统一由网关和 router 管理。

## 7. 安全与合规设计

### 7.1 安全红线落实
- 禁止敏感信息硬编码：全部走环境变量 + 密钥管理。
- 敏感日志脱敏：手机号、邮箱、证件号掩码存储。
- 所有受保护接口都做鉴权和权限校验（RBAC + 数据权限）。
- 文件上传白名单：类型和大小限制，且扫描后入库。
- 生产前执行 SAST/DAST，阻断严重/致命漏洞发布。

### 7.2 隐私保护
- 最小化采集：仅采集问卷必要字段。
- 匿名问卷默认不采集真实身份信息。
- 数据留存策略可配置（按项目和法规要求）。

### 7.3 审计与追踪
- 全链路 `traceId`。
- 登录、发布、导出、权限变更全量审计。
- 日志可追溯并满足留存要求。

## 8. 反作弊与防重复方案

组合策略（可按问卷配置启用）：
- 账号维度：同账号提交次数限制。
- 设备维度：浏览器指纹 + 本地设备标识。
- 网络维度：IP 频控与黑名单。
- 批次维度：批次码有效性与次数校验。
- 行为维度：异常提交速度检测（过快提交预警）。

说明：因规范禁止前端直接操作 cookie，前端标识采用本地存储 token + 后端签名校验机制，不在前端代码层面进行 cookie 读写逻辑。

## 9. 本地部署与云化演进

### 9.1 本地运行（第一阶段）
- 部署方式：Docker Compose。
- 组件：gateway + 6 个业务服务 + PostgreSQL + Redis + Kafka + MinIO。
- 登录：演示账号模式，支持后续切换到 SSO。

### 9.2 云化部署（第二阶段）
- 容器化：Kubernetes（Deployment + HPA）。
- 网关与入口：ALB/Ingress。
- 配置中心：Apollo 或 Spring Cloud Config。
- 可观测：SkyWalking + Prometheus + Grafana。
- 对象存储与消息：云原生 OSS/S3 + MQS/Kafka 托管。

### 9.3 环境策略
- `local`：开发联调与演示。
- `test`：集成测试与性能压测。
- `prod`：高可用与审计合规。

## 10. 权限与角色

- 系统管理员：租户配置、权限、审计、模板治理。
- 问卷管理员：创建问卷、发布投放、查看汇总。
- 分析员：查看统计报表、导出数据。
- 填写用户：按授权访问填写页（登录态或匿名）。

权限模型：
- RBAC（角色） + ABAC（数据范围，如仅能查看本部门问卷）。

## 11. 非功能性指标（NFR）

- 可用性：核心服务月度可用性 >= 99.9%（生产目标）。
- 性能：单问卷提交接口 P95 < 300ms（不含网络抖动）。
- 扩展性：问卷并发填写峰值支持横向扩容。
- 可维护性：服务、模块、接口、日志统一规范。

## 12. 测试与验收标准

### 12.1 测试范围
- 单元测试：题型解析、逻辑跳转、反作弊规则。
- 集成测试：发布-填写-统计-导出全流程。
- 安全测试：鉴权、越权、注入、上传、敏感信息泄漏。
- 性能测试：高并发提交与报表导出。

### 12.2 业务验收用例（关键）
- 可创建包含全部题型的问卷并成功发布。
- 支持匿名与登录态填写，跳题逻辑正确生效。
- 截止时间与次数限制准确拦截。
- 重复提交能被策略拦截并记录原因。
- 可生成 Excel/CSV 导出且字段完整。
- 统计图表与原始数据一致。
- 简洁模式切换后填写流程正常。

## 13. 分阶段实施计划

### 阶段 A：方案与骨架（1-2 周）
- 完成详细设计、数据模型、接口定义。
- 搭建微服务骨架与本地基础组件。

### 阶段 B：核心能力（2-4 周）
- 问卷设计、题型引擎、发布与填写、基础反作弊。

### 阶段 C：增强能力（2-3 周）
- 客户管理、导入导出、统计图表、审计与权限完善。

### 阶段 D：云化准备（1-2 周）
- 容器化、监控告警、配置中心、灰度发布预案。

## 14. 风险与应对

- 跳题逻辑复杂导致维护成本高：采用可视化规则与发布前校验。
- 统计性能瓶颈：热数据缓存 + 异步聚合 + OLAP 预留。
- 合规风险：上线前执行安全清单和隐私影响评估。
- 导入数据质量问题：模板校验 + 错误回传 + 可重试机制。
